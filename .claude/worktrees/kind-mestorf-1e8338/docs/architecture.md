# SysOps Hub — Arquitectura

## Resumen

SysOps Hub es una aplicación web interna para equipos de sistemas. Centraliza la operación diaria:
estado de infraestructura, incidencias, automatizaciones, base de conocimiento y auditoría.

## Capa de Conectores

Cada integración implementa la interfaz `BaseConnector`:

```typescript
abstract class BaseConnector {
  abstract healthCheck(): Promise<ConnectorHealth>
  abstract getAlerts():   Promise<AlertSummary[]>
  abstract getSystems():  Promise<SystemStatus[]>
}
```

**Conectores disponibles:**

| Conector     | Implementación | Notas                          |
|--------------|---------------|-------------------------------|
| Zabbix       | Real (JSON-RPC 2.0) | Auth token con caché 55min |
| Uptime Kuma  | Real (REST API)     | Requiere API key           |
| Proxmox      | Real (REST API)     | Ticket auth, SSL opcional  |
| vCenter      | Real (REST API)     | Session-based auth         |
| Portainer    | Real (REST API)     | API key                    |
| NAS/Synology | Real (DSM API)      | SID auth                   |
| Mock         | Demo              | Datos ficticios realistas  |

Si `DEMO_MODE=true`, todos usan `MockConnector` con datos ficticios.

Si un conector falla, el `ConnectorRegistry` captura el error y continúa con el resto.
La UI muestra el conector como "degradado" con el mensaje de error.

## Flujo de Datos

### Ciclo de alertas
```
Cron (cada 2min)
  → registry.checkAll()
  → connector.getAlerts()  [paralelo, con timeout 5s]
  → persist connector_status in DB
  → Dashboard poll (cada 60s) trae datos frescos
```

### Cierre de incidencia → KB
```
POST /incidents/:id/close { rootCause, resolution }
  → UPDATE incidents SET status='cerrada', rootCause, resolution
  → INSERT kb_articles (autoGenerated=true)
  → UPDATE incidents SET kbArticleId
  → INSERT audit_events
  → sendTeamsAlert (si Teams configurado)
```

### Ejecución de automatización
```
POST /automations/actions/:id/run
  → INSERT automation_runs (status=pendiente)
  → automationQueue.add(job)
  → Worker executa job asíncrono
  → UPDATE automation_runs (output, status)
  → Frontend polls /runs/:id cada 2s
```

## Seguridad

- **Auth**: JWT (HS256) con expiración configurable (default 8h). Tokens en localStorage.
- **Roles**: `admin > tecnico > readonly`. Cada endpoint y acción verifica el rol mínimo.
- **Rate limiting**: 200 req/min por IP (ajustable).
- **CORS**: Solo orígenes permitidos explícitamente.
- **Helmet**: Headers de seguridad estándar.
- **Audit trail**: Toda acción que modifica estado queda en `audit_events`.
- **Passwords**: bcrypt con salt factor 12.
- **Entra ID**: Preparado — activar con variables `ENTRA_*`. Requiere adapter OIDC.

## Base de Datos

Esquema relacional en PostgreSQL con Drizzle ORM.
Migraciones generadas con `drizzle-kit`.

Tablas principales:
- `users`, `systems`, `alerts`
- `incidents`, `checklist_items`, `incident_comments`, `incident_alerts`
- `automation_actions`, `automation_runs`
- `kb_articles`
- `audit_events`
- `connector_status`

## Decisiones y Tradeoffs

| Decisión | Alternativa | Razón |
|---------|------------|-------|
| Fastify | Express | Más rápido, TypeScript nativo, validación integrada |
| Drizzle | Prisma | Más ligero, SQL cercano, mejor TS inference |
| BullMQ | node-cron puro | Persistencia, reintentos, concurrencia, historial |
| React Query | SWR | Más completo, mutations, devtools |
| Zustand | Redux | Mínimo boilerplate para estado simple |
| PostgreSQL | MongoDB | Audit trail requiere transacciones y relaciones |
| JWT local | Session cookies | Más fácil para arquitectura stateless en LAN |
